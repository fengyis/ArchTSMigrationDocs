# 安卓到鸿蒙代码迁移：从 Skills 驱动到多 Agent 系统工程

---

## 一、当前 Skills 驱动方案面临的结构性问题

在 Android → HarmonyOS / ArkTS 代码迁移场景下，前期采用 **Skills 驱动方案**，将迁移知识、步骤、规则和修复经验外置为 skills，由大模型按需调用完成任务。这一方案在早期穿刺阶段非常有效——快速打通了迁移链路，积累了大量领域语料、错误样本和技能原型。

但随着迁移链路拉长、Skill 数量从个位数增长到数十个，这一方案暴露出三个结构性问题：

### 1. Skill 依赖链越来越复杂

Skills 不再是孤立能力，而是形成了大量依赖、引用和链式调用。例如 `a2h-spec` 依赖多个领域 skills，`a2h-execute` 依赖多个执行与修复相关 skills，不同 skills 之间还存在相互调用和补充关系。当 skill 数量增加时，依赖不透明、组合路径复杂、变更影响面扩大，演进成本持续上升。

### 2. Skill 数量增多导致模型遵从度下降

随着 skillbank 不断膨胀，大模型面对大量可选能力时，容易出现 skill 错选、漏选、使用不完整，以及对 skill 指令遵从度下降。**Skills 越多，并不天然意味着系统越强，反而因为上下文负担和选择复杂度上升导致效果下降。**

### 3. 链式调用使评测困难

当 skills 形成链式调用后，某个结果变差，难以判断是哪个 skill 退化；单一 skill 的效果会被上下游 skill 干扰；单独评测一个 skill 与真实链路效果可能不一致。评测从单点问题变成了系统性问题。

### 小结

Skills 驱动非常适合前期穿刺和能力积累，但其天然问题决定了：**它不适合作为长期唯一的工程架构。** 随着 skill 数量和链路复杂度上升，系统会逐渐出现依赖复杂、遵从度下降、评测困难等规模化失控现象。

---

## 二、内网环境约束

除了上述技术层面的结构性问题，还存在一个现实约束：**Claude Code 在内网环境下不可用。**

当前 skills 驱动方案运行在 Claude / Opus 强模型上，依赖外网调用。但实际生产部署必须面向内网环境，因此不能将 Claude Code 作为唯一交付路线。这一约束决定了项目必须建立一条可在内网独立运行的技术主线。

---

## 三、方案：双轨并行演进

基于上述问题分析和环境约束，提出以下方案：

### 路线 A：Claude Code 上持续摸高（Skills 驱动）

保留当前基于 Claude / Opus 最强模型能力的 **skills 驱动方案**，继续演进并摸高效果上限。

这条路线的定位是：
- 作为当前最强模型条件下的**效果天花板参考**
- 作为 OpenCode 路线的 **benchmark**
- 为后续系统演进提供客观基线和对照组

### 路线 B：在 **OpenCode** 上构建面向百万行代码迁移的 **多 Agent 工程化架构**。

OpenCode 上构建多 Agent 架构（主线）

这条路线的目标是建设一套：
- **可控**：系统职责清晰，问题可定位
- **可扩展**：从小规模样本到百万行代码平滑演进
- **可演进**：能力可独立优化，经验可自动沉淀
- **可内网部署**：不依赖外部模型服务

> 从架构上看，这是从 Phase 1（skills 驱动穿刺）到 Phase 2（主从 Multi-Agent 工程化演进）的自然升级。Skills 不消失，而是下沉为 Agent 的能力底座，由 Skill Router 统一调度。

## 五、多 Agent 架构重点解决的核心问题

这套多 Agent 架构不是为了"架构升级"而升级，而是直接面向当前迁移系统中的五个核心挑战：

### 挑战一：Spec 完备度

大规模代码仓中功能树层级深、UI/UX 隐式表达、功能与实现强耦合，导致 Spec 生成存在遗漏、幻觉和稳定性差的问题。Spec 是整个迁移链路的上游输入，**Spec 质量的天花板，就是迁移系统功能完备度的天花板。**

→ 多 Agent 架构中，**Spec Generator** 作为独立角色，可以围绕 spec 质量做独立评测和优化，不再混杂在代码生成流程中。同时引入 **Spec Audit Agent** 进行规范检查和完备度评估。

### 挑战二：测试反馈闭环和问题定位定界

编译通过不等于功能可用。当前缺少系统性的验证 → 定位 → 修复 → 回归链路，尤其是**问题定位定界**——出了问题不知道是 Spec 层错误、代码逻辑错误、还是平台适配问题。

→ 多 Agent 架构中，**QA / Verify Agent** 专职验证与问题回流，能结构化地将问题反馈给上游，形成 `生成 → 验证 → 问题定位定界 → 修复 → 回归验证` 的完整闭环。

### 挑战三：Skill 优化演进

随着 skills 从前期穿刺工具成长为系统核心能力底座，面临两个子挑战：

**子挑战 1：Skill 规模化选择**

Skill 数量增长后出现错选、漏选、重复加载、依赖缺失和上下文失控问题。Host Agent 自主选 Skill 的准确率随数量增加显著下降，不相关 Skill 被加载进 context 造成上下文膨胀，进一步降低模型遵从度。

→ 多 Agent 架构中引入 **Skill Router**，废弃全局全量注入，改为基于角色的按需路由——PM Agent 只访问分析型 skills，Coder 只拉取代码生成相关技能包，QA 专用验证与修复技能包。Skills 下沉为底层设施。

**子挑战 2：Skill 评测**

当 skills 形成链式调用后，评测从单点问题变成系统性问题：某个结果变差，难以判断是哪个 skill 退化；单一 skill 的效果会被上下游 skill 干扰；单独评测一个 skill 与真实链路效果可能不一致。缺乏统一评测体系，skill 的质量和演进方向就无法客观度量。

→ 建立 **Skill 评测体系**，覆盖从单 Skill 上传到 E2E 链路的全流程评测。通过标准化的评测方案，量化 Skill 命中率、依赖解析成功率和链路收益，为 Skill 的新增、优化和淘汰提供数据依据。

### 挑战四：单 Agent 承载瓶颈

单 Agent 的 context 有限，无法同时持有所有能力上下文；长链路任务容易中途断裂；系统可观测性差。

→ 多 Agent 架构通过**角色拆分**（PM / Spec Generator / Coder / QA）实现职责解耦。用 clear handoff 替代隐式 skill 串联，每个环节的输入输出可独立观测和评测。

### 挑战五：经验沉淀从人工触发走向系统化

当前系统并非完全没有经验回流能力——已有基于人工触发的 **retrospect skill**，可以在迁移任务结束后做反思总结，提取错误模式和改进建议。但这一机制存在明显局限：

- **依赖人工触发**：必须有人主动发起 retrospect，系统不会自动沉淀
- **结果散落**：反思产出的经验以文本形式存在，没有结构化入库，无法被后续任务自动检索和复用
- **覆盖不完整**：只有被人工关注到的任务才会做反思，大量执行轨迹和失败模式被遗漏
- **无法闭环**：即使产出了改进建议，也缺少自动将其反向作用于 Spec、Skill、Router 策略的机制

挑战在于：**如何将当前人工触发的 retrospect 能力，拓展为一套自动化的、可结构化沉淀和检索的、能反向驱动系统优化的长期演进机制。**

→ 多 Agent 架构预留 **Memory 演进层**，目标是将 retrospect 从"人工触发的一次性反思"升级为"系统自动采集 → 结构化沉淀 → 按需检索 → 反向驱动优化"的完整闭环，推动系统形成数据飞轮。

## 七、总结

本项目面向 **50万～100万代码规模** 的 Android → HarmonyOS / ArkTS 迁移，阶段目标是在低人工干预条件下实现**端到端功能完备度 80%**。

前期通过 Claude / Opus 上的 **Skills 驱动方案**快速穿刺，打通了迁移链路并积累了大量能力原型。但随着 Skill 数量增长和链路复杂化，系统面临依赖复杂、遵从度下降和评测困难的结构性问题。加之 **Claude Code 在内网环境下不可用**，项目必须建立可内网部署的技术主线。

因此，方案采用**双轨并行演进策略**：在 Claude Code 上继续以 skills 驱动方案摸高效果上限，作为 benchmark 和对照组；同时在 **OpenCode 上构建面向百万代码迁移的多 Agent 架构**，作为系统能力建设的主战场。

项目拆分为六大建设模块：**Spec 生成与可信、测试反馈闭环（含问题定位定界）、Skill 优化演进（Router + 评测）、Multi-Agent 编排与平台落地、Memory 长期演进、统一评测与跨平台对比**。每个模块解决一个明确的工程问题，有独立的负责人、交付物和验收标准，共同构成一个完整的 AI 系统工程——Spec 提供上游输入，测试闭环驱动质量提升，Skill Router 统一能力调度，Multi-Agent 架构承载工程化协作，Memory 推动系统长期自演进。

最终，通过**统一评测体系**贯穿全局，覆盖跨平台（OpenCode vs Claude Code）、跨架构（Multi-Agent vs 纯 Skill）、跨方案（Skill Router vs 直接 Skills）、跨阶段（Verify/Fix 前后、Spec 优化前后）的多维度对比，持续验证收益、指导路线选择，形成一套可控、可扩展、可持续演进的规模化迁移系统。
